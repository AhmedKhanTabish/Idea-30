Embedded CI/CD Pipeline Architect — VS Code Extension
A VS Code extension that lets embedded developers right-click a project folder and generate a production-ready CI/CD pipeline (GitHub Actions or GitLab CI) tailored to their MCU project.

Design Principles (Scalability)
Separation of Concerns — The extension UI layer is thin; all logic lives in a standalone core/ engine that can be reused by a CLI tool, web app, or other IDE later.
Strategy Pattern for Scanners — Each project type (PlatformIO, CMake, Arduino, ESP-IDF, Zephyr) is a pluggable scanner. Adding a new one means adding a single file — zero changes to existing code.
Template + AI Hybrid — Static YAML templates handle the deterministic parts (caching, checkout, artifact upload). The LLM fills in project-specific logic (env names, build commands, test commands). This keeps output reliable while still being "smart."
Provider Pattern for CI Targets — GitHub Actions and GitLab CI are separate "providers." Adding Bitbucket Pipelines, Azure DevOps, etc. later is one new file.
User Review Required
IMPORTANT

LLM API Key: The extension will need an API key to call an LLM (OpenAI, Anthropic, or local Ollama). The plan includes a settings UI where the user configures their key. Do you have a preferred LLM provider? I'll default to supporting OpenAI + Ollama (free local option) with the architecture open to adding more.

IMPORTANT

Offline Fallback: If no LLM key is configured, the extension will use a pure-template mode (no AI reasoning, just deterministic YAML generation from detected config). This means it still works without any AI — just less "smart." Is this acceptable, or do you want AI to be mandatory?

2. Project Scanner (Detection Engine)
[NEW] 
types.ts
typescript
// Key interfaces — this is the contract all scanners implement
export interface ProjectProfile {
  projectType: 'platformio' | 'cmake' | 'arduino' | 'esp-idf' | 'zephyr' | 'unknown';
  platform: string;           // e.g., 'espressif32', 'ststm32', 'atmelavr'
  board: string;              // e.g., 'esp32dev', 'nucleo_f446ze'
  framework: string;          // e.g., 'arduino', 'espidf', 'stm32cube'
  environments: string[];     // Multi-env support from platformio.ini
  libraries: string[];        // Detected library dependencies
  testFramework?: string;     // 'unity', 'googletest', 'none'
  buildSystem: string;        // 'pio', 'cmake', 'make', 'idf.py'
  sourceFiles: string[];      // Detected .c/.cpp/.ino files
  extraConfig: Record<string, any>; // Extensible metadata
}
export interface IProjectScanner {
  readonly name: string;
  readonly priority: number;   // Higher = checked first
  canScan(folderPath: string): Promise<boolean>;
  scan(folderPath: string): Promise<ProjectProfile>;
}
[NEW] 
scannerRegistry.ts
Maintains ordered list of scanners by priority
register(scanner) / scanProject(folderPath) — tries each scanner, returns first match
Scalability: New scanners just call registry.register(new ZephyrScanner())
[NEW] 
platformioScanner.ts
Checks for platformio.ini existence
Parses INI format to extract [env:*] sections
Extracts platform, board, framework, lib_deps, test_framework
Returns fully populated ProjectProfile
[NEW] 
cmakeScanner.ts
Checks for CMakeLists.txt
Regex-based parsing for project(), target_link_libraries, toolchain file references
Detects STM32 via arm-none-eabi toolchain references
Detects ESP-IDF via idf_component_register or $ENV{IDF_PATH}
[NEW] 
arduinoScanner.ts
Checks for .ino files
Scans #include statements for board-specific libraries (ESP8266WiFi, STM32, etc.)
Detects board from boards.txt or arduino.json if present
[NEW] 
espIdfScanner.ts
Checks for sdkconfig or sdkconfig.defaults
Parses CONFIG_IDF_TARGET to determine chip (esp32, esp32s2, esp32s3, esp32c3, etc.)
Detects component structure (components/ dir, idf_component.yml)
